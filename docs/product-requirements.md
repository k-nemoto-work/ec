# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**ReUse Market** - リユース品特化型ECサイト（DDD学習プロジェクト）

### プロダクトコンセプト
- **1点もの管理**: リユース品の性質上、同一商品の複数在庫が存在しない。商品ステータスによる厳密な在庫管理を実現する。
- **DDDによる設計**: ドメイン駆動設計を実践し、ビジネスルールをドメイン層に集約した保守性の高いコードを構築する。
- **クリーンアーキテクチャ**: 関心事の分離を徹底し、テスト容易性・拡張性を最大化する。

### プロダクトビジョン
DDDとクリーンアーキテクチャの実践的な学習を目的とした、リユース品（1点もの）を扱うECサイトを開発する。
Kotlin / Spring Boot 3 という現代的な技術スタックを使い、複雑なビジネスルールをドメインモデルとして正確に表現することを優先する。
将来的にはドメインイベントを活用した結果整合的な処理の実装も視野に入れ、段階的に設計を進化させる。

### 目的
- DDD（ドメイン駆動設計）の実践的なスキルを習得する
- クリーンアーキテクチャによる層の分離を実際のコードで体験する
- Kotlin / Spring Boot 3 / Exposed / PostgreSQL の技術スタックに習熟する
- リユース品ECという現実に即したドメインモデルを設計・実装する
- ポートフォリオとして公開できる品質のコードベースを構築する

---

## ターゲットユーザー

### プライマリーユーザー: 出品者兼管理者
- リユース品（古着、家電、雑貨など）を販売したい個人
- 商品の登録・公開・ステータス管理を行う
- 注文を受け付け、発送処理まで管理する

### セカンダリーユーザー: 購入者
- リユース品をオンラインで購入したい個人
- 商品を検索・閲覧し、カートに追加して注文する
- 注文履歴を確認し、配送状況を把握する

### 開発者ペルソナ（学習目的）
- **背景**: DDD・クリーンアーキテクチャを業務で活用したいバックエンドエンジニア
- **課題**: 設計論を学んでも実際のコードに落とし込む経験が少ない
- **期待**: ECサイトという身近なドメインで、集約・値オブジェクト・ドメインサービスを実際に実装したい
- **ゴール**: 設計判断の根拠を説明できるレベルのコードベースを完成させる

## 成功指標 (KPI)

### プライマリーKPI（学習・品質指標）
- ドメイン層のテストカバレッジ: 80%以上
- ビジネスルールのドメイン層への集約率: 100%（インフラ層にビジネスロジックを持ち込まない）
- OpenAPIドキュメントによる全APIエンドポイントの仕様化: 100%

### セカンダリーKPI（機能完成度）
- MVPスコープの全機能実装: 5集約（顧客・商品・カート・お気に入り・注文）すべて完了
- 統合テスト（TestContainers使用）によるAPIエンドポイントカバレッジ: 主要フロー100%

## 機能要件

### コア機能 (MVP)

#### 1. 顧客管理（会員登録・認証）

**ユーザーストーリー**:
購入者として、安全にサービスを利用するために、メールアドレスとパスワードで会員登録・ログインしたい

**受け入れ条件**:
- [ ] メールアドレスとパスワードで会員登録できる（メールアドレスはシステム全体でユニーク）
- [ ] 登録済みのメールアドレス・パスワードでログインし、JWTトークンを取得できる
- [ ] パスワードは8文字以上、英数字混在で設定する必要がある
- [ ] パスワードはBCryptでハッシュ化して保存される
- [ ] 非活性ステータスの顧客はログインできない
- [ ] 顧客は配送先住所（郵便番号・都道府県・市区町村・番地）を登録・更新できる

**優先度**: P0（必須）

---

#### 2. 商品カタログ管理

**ユーザーストーリー**:
出品者として、商品を登録・公開管理するために、商品情報のCRUDと公開ステータスの管理がしたい

**受け入れ条件**:
- [ ] 商品名・価格・説明・カテゴリを指定して商品を登録できる（商品名は1〜100文字、説明は2000文字以内）
- [ ] 商品ステータスを「販売中 / 売約済み / 売却済み / 非公開」で管理できる
- [ ] 購入者は「販売中」ステータスの商品のみ一覧・詳細を閲覧できる
- [ ] 商品をカテゴリで絞り込みできる
- [ ] 価格は0より大きい値でなければ登録できない

**優先度**: P0（必須）

---

#### 3. カート管理

**ユーザーストーリー**:
購入者として、購入したい商品を一時的にまとめるために、カートへの追加・削除機能が欲しい

**受け入れ条件**:
- [ ] 「販売中」ステータスの商品のみカートに追加できる
- [ ] 同一商品の重複追加はエラーになる
- [ ] 1人の顧客につきカートは1つのみ存在する
- [ ] カートから商品を削除できる
- [ ] カート内の商品一覧と合計金額を確認できる

**優先度**: P0（必須）

---

#### 4. お気に入り管理

**ユーザーストーリー**:
購入者として、気になる商品をあとで見返すために、お気に入りリストへの追加・削除機能が欲しい

**受け入れ条件**:
- [ ] 「販売中」または「売約済み」ステータスの商品をお気に入りに追加できる
- [ ] 「売却済み」または「非公開」の商品はお気に入りに追加できない
- [ ] 同一商品の重複追加はエラーになる
- [ ] 1人の顧客につきお気に入りリストは1つのみ存在する
- [ ] お気に入りリストから商品を削除できる

**優先度**: P0（必須）

---

#### 5. 注文管理

**ユーザーストーリー**:
購入者として、カートの商品を購入確定するために、注文・決済・配送の管理機能が欲しい

**受け入れ条件**:
- [ ] カート内の商品を注文確定できる
- [ ] 注文確定時に商品ステータスが「売約済み」に変更される（既に売約済みの場合はエラー）
- [ ] 注文明細には注文時点の商品名・価格がスナップショットとして保存される
- [ ] 注文確定後はカートの中身が空になる（カートオブジェクト自体は残る）
- [ ] 注文確定後は注文内容を変更できない
- [ ] 注文ステータスを「未確定 / 確定 / 配送中 / 配送完了 / キャンセル」で管理できる
- [ ] 配送中・配送完了後はキャンセルできない
- [ ] 決済ステータスが「決済済み」にならないと配送ステータスを進められない
- [ ] 決済・配送はモック実装（外部連携なし）
- [ ] 顧客は自分の注文履歴を一覧・詳細で確認できる

**優先度**: P0（必須）

---

### 将来的な機能 (Post-MVP)

#### ドメインイベントによる結果整合性

注文確定時のドメインイベント（`OrderPlaced`）発行と、それをトリガーとした在庫ステータス更新の非同期処理を実装する。

**優先度**: P2（できれば）

#### 商品検索・絞り込み強化

キーワード検索、価格帯フィルタ、並び替え機能の追加。

**優先度**: P2（できれば）

#### レビュー・評価機能

購入者が商品・出品者へのレビューを投稿できる機能。

**優先度**: P2（できれば）

## 非機能要件

### パフォーマンス
- API応答時間: 通常操作で200ms以内（ローカル環境）
- 商品一覧取得: 20件のページネーションを100ms以内で返す（デフォルトページサイズ）

### ユーザビリティ（API設計）
- OpenAPI（Swagger）でAPI仕様を自動生成・公開する
- エラーレスポンスは一貫した形式（エラーコード・メッセージ）で返す
- HTTPステータスコードをREST仕様に従い正確に使用する

### 信頼性
- ドメイン層のユニットテストカバレッジ 80%以上
- TestContainersを使用したDB統合テストで主要フローを検証
- Flyway によるDBマイグレーション管理（ロールバック可能な状態を維持）

### セキュリティ
- パスワードはBCryptでハッシュ化して保存
- JWT による認証（有効期限設定あり）
- 他の顧客のリソース（注文・カート・お気に入り）にはアクセスできない（認可制御）
- SQLインジェクション対策（Exposed の型安全DSLを使用）

### スケーラビリティ
- 個人開発スコープのため、現時点では水平スケールは考慮しない
- ドメイン層の疎結合設計により、将来的なマイクロサービス分割に対応できる構造を維持する

### 保守性
- ドメイン層はインフラ層・プレゼンテーション層に依存しない（依存の方向はドメイン向き）
- ビジネスルールはドメインオブジェクト（集約・値オブジェクト・ドメインサービス）に集約する
- 各層の役割が明確に分離されており、独立してテスト可能な構造を維持する

---

## スコープ外

以下の機能はこのプロジェクトのスコープ外とする:

- 実際の決済処理（クレジットカード、PayPayなど外部決済サービスとの連携）
- 返金処理（決済ステータスの `REFUNDED` 遷移は本スコープ外）
- 実際の配送業者との連携（追跡番号発行など）
- フロントエンド（WebUI / モバイルアプリ）の開発
- メール通知機能
- 管理者向けダッシュボード
- 画像アップロード・CDN連携
- 本番環境へのデプロイ・インフラ構築
- マルチテナント対応
